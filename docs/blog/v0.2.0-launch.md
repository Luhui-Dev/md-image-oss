# 写了三年技术博客后，我用 200 行 Python 把 Markdown 图床自动化了

> 开源地址：https://github.com/Luhui-Dev/md-image-oss
> 当前版本：v0.2.0，支持 Markdown / MDX / HTML

## 一、那个让我每次写完都想骂人的 15 分钟

每个写技术博客的人，多半都经历过这样一个流程。

写完一篇五千字的文章，截图二十张，路径全是 `./images/xxx.png`。要发布的时候，事情才刚刚开始。

打开掘金后台，一张张拖图，等上传，复制 URL，回到 Markdown 把本地路径替换掉。中间还要顺手拖到 TinyPNG 压一下，不然 2MB 一张的 macOS 截图直接干爆首屏加载。

我大致算过这个流程的成本：**单篇 15 分钟，全是机械操作**。如果同一篇文章要同步到知乎、个人博客、Notion，再乘以 N。

更要命的是几个隐性成本，平时不注意，攒着攒着就成了债：

1. **仓库越来越胖**：博客源码仓库里塞着几百张未压缩的 PNG，clone 一次几百兆
2. **跨平台搬家**：从掘金搬到自己博客，每张图都要重传，链接全断
3. **手动压缩劝退**：知道压缩很重要，但开 TinyPNG 网页太累，每次写完都告诉自己"下次"
4. **链接烂了**：用过的某个图床跑路了，三年前的文章一夜之间全是裂图

我忍了三年，最近终于决定写个工具自己用。结果发现这事看起来简单，写起来意外地有几个坑值得说说。今天把它开源了，叫 [md-image-oss](https://github.com/Luhui-Dev/md-image-oss)。

一句话能讲清楚的功能：**一行命令，把 Markdown / MDX / HTML 里的本地图片自动压缩、上传到阿里云 OSS，再把文档里的路径替换成 OSS URL**。

## 二、现成方案为什么都差一点意思

动手之前我也调研过一圈。不是没好工具，但都不解决我的问题。

- **PicGo**：经典 GUI 工具，可惜要写完文章再回头一张张拖，本质还是手动
- **Typora 内置上传**：好用，但绑死编辑器，我用 Cursor / VS Code 写就用不上
- **手动 ossutil**：要写 shell 脚本，还得自己改文档里的路径，没有人想第二次写这种胶水
- **图床 SaaS**：付费、限流、数据不在自己手里，跑路风险一直在

我想要的工具长这样：

> 一行命令，吃进 Markdown，吐出 Markdown。本地图片自动上 OSS，幂等可重入，能塞进 CI。

简单说：**写 → 跑一行命令 → 提交**。中间不需要打开浏览器，不需要打开第二个工具。

这事看上去 50 行 Python 加个正则就完了，但真动手才发现，里面藏着至少 4 个坑。

## 三、看起来 grep 一下就完事，实际有四个坑

### 坑 1：博客里教大家写 Markdown 的代码块

我的博客里经常有这样的段落：

````markdown
比如插一张图，可以这样写：

```markdown
![示例](./assets/demo.png)
```
````

这张 `demo.png` 是**示例代码的一部分**，不是真实图片引用。如果工具粗暴 grep `![...](...)`，会把示例代码里的路径也改成 OSS URL，README 当场炸——读者照抄你的"示例"，发现自己的 `./assets/demo.png` 怎么变成了一个 cdn 地址。

[md_image_oss/processor.py:67](md_image_oss/processor.py:67) 处理这个的方式不是用 AST 解析（Markdown 没标准 AST，自己写解析器又重），而是先做一次"代码区隔离"：

```python
# Fenced code blocks and inline code in Markdown.
_MD_CODE_RE = re.compile(
    r"```.*?```|~~~.*?~~~|`[^`\n]+`",
    re.DOTALL,
)
```

然后在 [processor.py:111](md_image_oss/processor.py:111) 用一个"切片重写"模式：把代码区先切出来原样保留，只在非代码区做替换：

```python
def _split_and_rewrite(self, content, base_dir, skip_re):
    out_parts, cursor = [], 0
    for m in skip_re.finditer(content):
        if m.start() > cursor:
            out_parts.append(self._rewrite(content[cursor:m.start()], base_dir))
        out_parts.append(m.group(0))   # 代码区原样保留
        cursor = m.end()
    if cursor < len(content):
        out_parts.append(self._rewrite(content[cursor:], base_dir))
    return "".join(out_parts)
```

简单，但可靠。

### 坑 2：MDX 里那个让无数工具翻车的 JSX

写 Next.js / Docusaurus / Astro 博客的人都用过 MDX，它允许在 Markdown 里嵌 JSX：

```jsx
<img src={dynamicSrc} />     // 这是个 JS 表达式，绝对不能动
<img src="./local.png" />    // 这才是要替换的
```

如果工具直接 `s/src=([^ ]+)/.../`，第一行的 `{dynamicSrc}` 会被识别成 URL，被当作"远程图片"处理，运行时直接 404。

[processor.py:48](md_image_oss/processor.py:48) 的处理是设计上的"明智之举"：

```python
# Only quoted string values; JSX expressions like src={foo} are left alone.
_HTML_SRC_RE = re.compile(r"""src\s*=\s*["']([^"']+)["']""", re.IGNORECASE)
```

只匹配带引号的字符串字面量。`{}` 表达式天然被跳过，不需要任何额外逻辑。这是个"少做就是多做"的设计——不显眼，但很关键。

### 坑 3：HTML 里的 script / style / pre / code

HTML 文档同样有"假图片"陷阱：

- `<script>` 里可能有图片 URL 字符串（`"image": "/banner.png"`），但那是 JSON 数据，不是渲染的图
- `<style>` 里有 `background: url(./bg.png)`，是 CSS 不是内容
- `<pre>` 和 `<code>` 用来展示代码，里面的 `<img>` 是给人看的"示例"

[processor.py:73](md_image_oss/processor.py:73) 用同一套"切片重写"模式处理：

```python
_HTML_SKIP_RE = re.compile(
    r"""
      <script\b[^>]*>.*?</script\s*>
    | <style\b[^>]*>.*?</style\s*>
    | <pre\b[^>]*>.*?</pre\s*>
    | <code\b[^>]*>.*?</code\s*>
    | <!--.*?-->
    """,
    re.DOTALL | re.IGNORECASE | re.VERBOSE,
)
```

`MarkdownProcessor` 用 `_MD_CODE_RE` 切片，`HtmlProcessor` 用 `_HTML_SKIP_RE` 切片，**逻辑共用同一个 `_split_and_rewrite`**。这是用代码结构去匹配问题结构。

### 坑 4：被九成工具忘了的引用式定义

Markdown 还有一种引用式语法，写长文档时用来集中管理 URL：

```markdown
看一下这张图：![logo][hero]

后面其它地方也用：![另一处][hero]

[hero]: ./assets/hero.png "封面图"
```

`![logo][hero]` 不是图片定义，定义在文件底部的 `[hero]: ./assets/hero.png` 才是。要替换路径不能动 `![logo][hero]` 那一行，要去最后那行改路径。

[processor.py:52](md_image_oss/processor.py:52) 单独写了一个 `_REF_DEF_RE` 处理这种情况，[processor.py:246](md_image_oss/processor.py:246) 的 `_rewrite_references` 逐行匹配并保留原行尾符号。这个语法多数工具直接忽略，对不信邪的我必须支持。

> 这四个坑放一起看，会发现都不是"算法难"，而是"每个边界都要想清楚"。Markdown 看似简单，做工具时其实是个不小的状态机。

## 四、几个值得说说的设计决策

下面这三个是我在这个项目里觉得**值得拿出来聊**的取舍。每个都有代价，但代价都可接受。

### 决策 1：用 SHA256 当文件名，而不是保留原名或随机 UUID

[md_image_oss/uploader.py:35](md_image_oss/uploader.py:35)：

```python
digest = hashlib.sha256(data).hexdigest()[:24]
filename = f"{digest}{ext}"
key = f"{self.config.prefix}/{filename}" if self.config.prefix else filename

if not self.bucket.object_exists(key):
    self.bucket.put_object(key, data, headers=headers)
```

24 个十六进制字符的前缀，对应 96 bits 熵——一个 bucket 里全是 24 字符的随机串，发生碰撞需要约 `2^48` 张图，平均超过两亿亿张，可以认为不会碰撞。

这个选择带来三个好处：

- **天然去重**：同一张截图被三篇文章引用，物理上只存一份。我把 macOS 那张默认壁纸从我的几十篇笔记里拖进来过几十次，bucket 里只有一份
- **修改即新版本**：图改了，哈希就变，老 URL 还在，老文章不会受影响
- **路径不再碰中文 / 空格 / emoji**：用户原文件名是 `截图 2024-12-01 14.32.png`？无所谓，对外是哈希

代价是上传前必须读完整文件算哈希。但图片本来就要读取压缩，哈希顺手做了，几乎零成本。再加一个 `bucket.object_exists()` 预检查，重复上传连网络都省了。

### 决策 2：压完比原图大就回滚

JPEG 压缩有个反直觉的现象：**对一张已经被高度压过的图，再压一遍可能更大**。原因是再次编码会引入新的 DCT 噪声，而 quality=85 的最低保底会守住质量底线。

所以 [md_image_oss/compressor.py:100](md_image_oss/compressor.py:100) 三个编码器都做了同一件事：

```python
out = io.BytesIO()
img.save(out, format="JPEG", quality=quality, optimize=True,
         progressive=True, subsampling=2)
encoded = out.getvalue()
if len(encoded) < len(original):
    return encoded, ".jpg"
return original, ".jpg"   # 压完更大，返回原图
```

PNG 同理，WebP 同理。再加一条规则：**GIF 和 SVG 一律不动**——压 GIF 容易破坏动画帧，压 SVG 没意义还可能丢属性。

跑一次的实际效果（README 里有个例子）：

```text
✓ ./images/cover.png
    → https://cdn.example.com/markdown/8b3f9e6a4c1d2e7f9a0b.jpg
    482.1 KB → 138.4 KB  (-71%)
✓ ./images/map.jpg
    → https://cdn.example.com/markdown/3a1c4d5e6f7081929394.jpg
    1.2 MB → 410.5 KB  (-67%)
```

两张图省了 1.1 MB，全自动。

### 决策 3：密钥全走环境变量，不进配置文件

[md_image_oss/config.py:31](md_image_oss/config.py:31) 没有 YAML、没有 TOML，就一个 `from_env()`：

```python
@classmethod
def from_env(cls) -> "Config":
    missing = [name for name in REQUIRED_VARS if not os.getenv(name)]
    if missing:
        raise EnvironmentError("Missing required environment variables: ...")
    return cls(
        access_key_id=os.environ["OSS_ACCESS_KEY_ID"],
        access_key_secret=os.environ["OSS_ACCESS_KEY_SECRET"],
        ...
    )
```

为什么不做配置文件？做配置文件最大的风险是用户提交进 Git。一旦 AccessKey 进了公开仓库，哪怕立刻删，git history 里它也活着，必须重新生成。

环境变量的好处是**物理隔离**：

- 本地开发：`.env` 进 `.gitignore`，工具会读，但不会跟着 commit 走
- CI/CD：直接用 GitHub Actions / GitLab Runner 的 secrets 注入，没有任何中间文件
- 生产服务器：用 1Password CLI / pass / Vault 注入，运行结束自动消失

代价是对完全没接触过环境变量的新手有一点学习曲线。但只要做这件事的人愿意接触阿里云，这点学习成本是值得的。

## 五、十秒上手

文章看到这里，想试一下的话整个流程是这样的。

### 安装

目前还没发到 PyPI（在路线图里），先用 git clone：

```bash
git clone https://github.com/Luhui-Dev/md-image-oss
cd md-image-oss
pip install -e .
```

需要 Python 3.9+。

### 配置

把 `.env.example` 复制成 `.env`，填四个必填变量：

```bash
OSS_ACCESS_KEY_ID=...
OSS_ACCESS_KEY_SECRET=...
OSS_ENDPOINT=https://oss-cn-hangzhou.aliyuncs.com
OSS_BUCKET=my-blog
# 可选
OSS_PREFIX=markdown
OSS_CUSTOM_DOMAIN=cdn.example.com
```

强烈建议给这个工具创建一个**专属 RAM 子账号**，权限只给目标 Bucket 的 `oss:PutObject` / `oss:GetObject`。出问题也只是这一个 bucket 受影响，不会满盘皆输。

### 跑

```bash
md-oss article.md --env-file .env
```

输出 `article.oss.md`，本地图片全部替换成 OSS URL。

### 几个最常用姿势

```bash
md-oss article.md -i              # 原地覆盖，写完即发
md-oss article.md -q 90           # 调整压缩质量
md-oss article.md --process-remote # 把别人博客的图也搬到自己 OSS
md-oss article.md --dry-run > preview.md  # 预览不写盘
```

`--process-remote` 这个特别提一下：写文章引用别人博客的图时，最怕对方哪天博客挂了或者裂图。加上这个参数，工具会下载远程图、压缩、上传到你自己的 OSS，**从此不再依赖别人的图床存活**。

### 塞进 GitHub Actions

最后一个用法是我自己最常用的——在 CI 里跑：

```yaml
name: Publish
on:
  push:
    paths: ['posts/**.md']
jobs:
  upload:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.11' }
      - run: pip install git+https://github.com/Luhui-Dev/md-image-oss
      - run: |
          for f in posts/*.md; do
            md-oss "$f" -i
          done
        env:
          OSS_ACCESS_KEY_ID: ${{ secrets.OSS_AK }}
          OSS_ACCESS_KEY_SECRET: ${{ secrets.OSS_SK }}
          OSS_ENDPOINT: ${{ secrets.OSS_ENDPOINT }}
          OSS_BUCKET: ${{ secrets.OSS_BUCKET }}
      - uses: stefanzweifel/git-auto-commit-action@v5
        with:
          commit_message: 'chore: upload images to OSS'
```

写完文章 push 上去，CI 自动跑工具、改路径、提交回来。从此发布只剩 `git push`。
